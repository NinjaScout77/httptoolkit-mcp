## 2. Auth token mechanism — MAJOR DEVIATION FROM PLANNING.md

### What PLANNING.md assumed
- A file-based auth token at OS-specific paths (`~/Library/Application Support/httptoolkit/auth-token`, etc.)

### What actually exists
- **No file-based token.** The auth token is ephemeral, generated by the Electron desktop app at startup and passed to the server via `HTK_SERVER_TOKEN` environment variable.
- The server accepts auth from: `HTK_SERVER_TOKEN` env var OR `--token` CLI flag.
- The desktop app deletes the env var from the server process after reading it (`delete process.env.HTK_SERVER_TOKEN`), so it cannot be recovered from the running process.
- The token is stored in Electron IPC (`ipcMain.handle('get-server-auth-token', ...)`) and is only accessible from within the Electron renderer process.

### Two communication channels discovered

1. **Unix domain socket** (`$TMPDIR/httptoolkit-ctl.sock` on macOS):
   - **No auth required**
   - Serves: `/api/status`, `/api/operations`, `/api/execute`
   - This is how the built-in MCP communicates with the server
   - Windows: `\\.\pipe\httptoolkit-ctl`
   - Linux: `$XDG_RUNTIME_DIR/httptoolkit-ctl.sock` or `/tmp/httptoolkit-<uid>/httptoolkit-ctl.sock`

2. **HTTP REST API** (port 45457):
   - **Requires auth token** (Bearer in Authorization header) and valid Origin header
   - Serves: all of the above PLUS `/client/send`, `/interceptors/:id/activate`, `/version`, `/config`, etc.
   - `/client/send` is only available here, not on the socket

### Implications for our architecture

**Read tools** (events_list, events_get, events_body, server_status, interceptors_list) can all work through the Unix socket via `/api/execute` — **no auth needed**.

**Replay tools** (replay_request, replay_raw) need `/client/send` which is HTTP-only — **auth required**.

### Recommended approach (Options for Pradeep)

**Option A: Dual transport (socket for reads, HTTP for writes)**
- Use Unix socket for all read operations (zero config needed)
- Use HTTP port with auth token for `/client/send` (user must set `HTK_SERVER_TOKEN`)
- Best UX for read-only usage, graceful degradation for replay

**Option B: HTTP-only with required token**
- Use HTTP port for everything
- User must always set `HTK_SERVER_TOKEN`
- Simpler code, but worse UX (can't even list events without config)

**Option C: Socket-only with no replay (Phase 1 scope change)**
- Only use the socket, defer replay to Phase 2 when we can explore alternatives
- Reduces Phase 1 to 5 read tools + server_status + interceptors_list

**Recommendation: Option A.** Most users will start with read-only exploration. Replay users accept more setup.

## 5. /client/send (HTTP-only, auth required)

Not verified directly (requires auth token). From source code analysis:

- `POST /client/send` on port 45457
- **Request body**: `rawBody` must be **base64-encoded** (differs from PLANNING.md which assumed raw string)
- **Response**: NDJSON stream (content-type: `application/x-ndjson`)
- **Response body parts**: `rawBody` field is **base64-encoded**
- Event types in stream: `request-start`, `response-head`, `response-body-part`, `response-end`, and error events
- Supports `options.proxyConfig` for upstream proxy
- Supports `options.clientCertificate` (pfx as base64)
- Abort via client disconnect

## 6. Tier exposure

`UiOperationBridge.isPaidUser()` exists and checks `channel.user.isPaidUser()`.
This comes from the WebSocket auth flow where the UI sends JWT account data.
It is NOT exposed via any REST endpoint or operation.

For `server_status`, tier will be `"unknown"` unless we find another way.

Operations have a `tiers` array (e.g., `["free", "pro"]`), indicating which tiers can use them. `interceptors.activate` is pro-only.

## 7. Binary body handling

From source code, response body parts use base64 encoding:
```javascript
if (evt.type === 'response-body-part') {
    res.write(JSON.stringify({
        ...evt,
        rawBody: evt.rawBody.toString('base64')
    }));
}
```

For `events.get-request-body` / `events.get-response-body`, the bodies come through `/api/execute` → the UI bridge → the Electron UI. The UI operations handle body encoding. Based on the output schema, bodies are returned as text strings with `totalSize` and `isTruncated` fields. Binary content handling needs further testing with actual binary captures.
